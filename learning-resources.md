## 🧠 思维模式与问题解决框架

### 1. 系统性调试思维

#### 问题分析框架
```markdown
**您的调试模式分析:**
1. **现象观察**: 具体描述问题表现（如"管道提前关闭"）
2. **范围界定**: 确定影响边界（如"大输入管道"）
3. **假设生成**: 基于系统知识提出可能原因
4. **实验验证**: 设计最小化测试验证假设
5. **根本原因**: 确认问题根源（如"下游命令提前退出"）
6. **解决方案**: 设计优雅的修复方案
7. **预防措施**: 添加防护避免重现

**实际案例: WinuxCmd cat命令修复**
- 问题: cat在大输入管道中不提前停止
- 分析: 下游可能提前关闭管道，但cat继续处理
- 方案: 添加管道状态检测，发现关闭立即退出
- 实现: `if (is_stdout_pipe_closed()) break;`
```

#### 性能分析思维
```markdown
**您的性能优化模式:**
1. **基准建立**: 确定当前性能水平
2. **热点识别**: 使用工具定位性能瓶颈
3. **方案设计**: 基于硬件特性设计优化
4. **实施优化**: 代码级别性能改进
5. **效果验证**: 量化优化效果
6. **权衡评估**: 评估优化代价和收益

**实际案例: AstraDB DashMap优化**
- 现状: 16个分片，每个分片有mutex锁
- 观察: NO SHARING架构下，Worker独占访问
- 优化: 移除分片锁，简化实现
- 结果: 代码减少18%，延迟改善3-4%
```

#### 技术决策框架
```markdown
**您的技术选型考虑因素:**
1. **性能影响** (40%权重): 执行效率、内存使用、并发能力
2. **代码质量** (30%权重): 可维护性、可读性、可测试性
3. **兼容性** (20%权重): 平台兼容、版本兼容、标准符合
4. **开发效率** (10%权重): 实现复杂度、工具支持、学习成本

**实际决策案例: snprintf vs wostringstream**
- 性能: snprintf更高效（栈分配 vs 堆分配）
- 质量: 两者都可读，snprintf更简洁
- 兼容性: 两者都标准，snprintf更通用
- 效率: snprintf实现更简单
- 决策: 选择snprintf进行优化
```
